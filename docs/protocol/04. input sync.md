# 04. Input Sync

The primary purpose of MouseWithoutBorders is to synchronize keyboard and mouse events across the network with zero perceived latency. To achieve this, it bypasses standard serialization and hooks directly into the OS's low-level input APIs.

## 1. Keyboard Sync
*(Source Reference: `App/Core/Event.cs`, `App/Core/InputSimulation.cs`)*

When the Host system's `InputHook` intercepts a key press, it drops the event from the local OS and routes it to the Controlled Machine.

### Payload Generation
- The packet is assigned `Type = PackageType.Keyboard` (`122`).
- The payload maps exactly to the `16-31` byte region overlapping the `KEYBDDATA` fields (Virtual Key Code `wVk`, state flags `dwFlags`, and `DateTime` ticks).

### Throttling and Ordering
Because TCP streams are continuous, rapid sequences of key events can sometimes overlap or flood the receiver's OS input queue. To guarantee correct evaluation of modifier keys (like Shift or Ctrl), the sender explicitly calls `Thread.Sleep(10)` immediately after sending a `KEYUP` or `SYSKEYUP` event. This 10ms micro-delay ensures the remote OS has fully processed the key release before the next key press arrives.

### Hotkey Interception
The Receiver actively inspects incoming keyboard payloads for specific OS-level sequences before passing them to the simulation layer:
1. **CTRL + ALT + DEL**: Cannot be injected normally in modern Windows. The protocol intercepts this (typically sent via `Delete` or `End` while modifiers are held down) and delegates it to an elevated Windows Service helper.
2. **Win + L**: The receiver intercepts this specific sequence (`L` + `WinDown`) and invokes `NativeMethods.LockWorkStation()` directly to lock the remote computer.
3. **Modifier State Resets**: If the machine loses connection or hits a transition boundary while keys are held down, the receiver artificially injects `KEYUP` events for all modifier keys (Win, Ctrl, Alt, Shift) to prevent "sticky keys."

## 2. Mouse Sync
*(Source Reference: `App/Core/Receiver.cs`, `App/Class/InputSimulation.cs`)*

Mouse synchronization dynamically transitions between Relative and Absolute coordinate systems to support different monitor sizes and DPI scaling across machines.

### Relative Movement (The `100,000` Offset)
For general mouse movement (`WM_MOUSEMOVE`), the protocol prefers sending raw Relative Deltas (`dx`, `dy`) rather than absolute positions. This preserves native mouse acceleration and smoothness on the remote machine.
To distinguish relative deltas from absolute coordinates, the sender applies a massive constant offset: `Event.MOVE_MOUSE_RELATIVE = 100000`.
- If `dx > 0`, it sends `dx + 100000`.
- If `dx < 0`, it sends `dx - 100000`.

### Absolute Positioning Scaling
When a click event occurs (e.g., `WM_LBUTTONDOWN`) or the mouse moves absolutely, the receiver uses a highly precise scaling factor across a 65535 coordinate grid.
1. **Host Side Mapping**: `(x - primaryScreenBounds.Left) * 65535 / screenWidth`.
2. **Client Side Restoration (`InputSimulation.cs`)**: The receiver calculates `dx = (md.X * w65535 / 65535) + l65535` and injects an `ABSOLUTE` mouse event.

*(Note: In certain internal jumps, such as `SwitchLocation`, a third constant `Event.XY_BY_PIXEL = 300000` is used to represent pixel-perfect leaps).*

### Absolute Fallback for Clicks
If the packet arrives in "Relative Mode" (values > 100,000) but it is a Click instead of a Move, the Receiver immediately ignores the relative offset, queries its own *local* cursor position (`NativeMethods.GetCursorPos`), converts that to a `0-65535` normalized absolute coordinate, and injects the click at that exact spot.

```csharp
if (Math.Abs(package.Md.X) >= 100000 && package.Md.dwFlags != WM.WM_MOUSEMOVE) {
    // We received a click while in relative mode! Convert to absolute:
    _ = NativeMethods.GetCursorPos(ref lastXY);
    package.Md.X = lastXY.X * 65535 / Common.screenWidth;
    package.Md.Y = lastXY.Y * 65535 / Common.screenHeight;
    _ = InputSimulation.SendMouse(package.Md);
}
```

## 3. Delegated Edge Detection & The `NextMachine` Packet
*(Source Reference: `App/Core/Receiver.cs`)*

The most complex and vital mechanism of Mouse Without Borders is how it determines when the cursor crosses over to a different machine.

**Crucial Architecture Requirement**: The Host Machine (the one with the physical mouse) is completely blind to where the cursor is on the remote screen. It relies on the remote machine to tell it when a boundary is hit!

### The Delegation Loop
1. The Host Machine sends relative movement deltas (`X+100000, Y+100000`).
2. The Controlled Machine receives the deltas and moves its local cursor via OS APIs.
3. Immediately after moving the cursor, the Controlled Machine checks if its own cursor just hit the physical boundary of its screen array (`MachineStuff.MoveToMyNeighbourIfNeeded`).
4. If it hits an edge, the Controlled Machine calculates which physical machine borders that edge (based on its own `MachinePool` Matrix), and calculates the entry coordinate (`0-65535`) for the new screen.
5. The Controlled Machine fires a `PackageType.NextMachine` (`121`) packet back to the Host!

### The `NextMachine` Hijacked Payload
To save on memory allocation, the `NextMachine` packet actually hijacks the standard 32-byte `Mouse Payload` (`MOUSEDATA`) structure:
- `X` and `Y`: The 65535-normalized entry coordinates for the new machine.
- `WheelDelta`: The integer `ID` (1 to 4) of the machine the Host needs to switch control to.

When the Host receives the `NextMachine` packet, it extracts the destination ID from `WheelDelta`, releases control of the current machine, updates its internal `Des` (Destination) pointer, and immediately begins routing all future input packets to the new machine!
