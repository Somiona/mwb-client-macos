# 08. Host Client Architecture & Roles

A common misconception when reimplementing KVM software is assuming a strict Client-Server architecture. MouseWithoutBorders, however, utilizes a **Symmetric Peer-to-Peer Mesh**. Every machine in the topology acts as both a Server and a Client simultaneously.

Furthermore, building a real-world client requires replicating the application's runtime lifecycle. The reference implementation relies on a robust set of thread loops, state monitors, and OS hooks to ensure seamless connectivity and cursor handoffs without blocking the primary input hooks.

This document serves as the ground-truth blueprint for the runtime architecture and peer-to-peer behavior of a host client.

## 1. Dynamic Role Switching
*(Source Reference: `App/Core/Event.cs`)*

There is no permanent "Host". The roles are entirely ephemeral and determined by physical interaction.

- **The Host (Controller)**: The machine where the user is currently physically moving the mouse or typing on the keyboard. When the cursor crosses a screen boundary, this machine begins routing its physical inputs over the network.
- **The Client (Controlled)**: The machine whose ID is currently assigned as the `Des` (Destination) field in the `Keyboard` and `Mouse` packets being transmitted by the Host.

If you drag the mouse from your Mac to a Windows PC, the Mac is the Host and the PC is the Client. If you immediately reach over and touch the physical mouse plugged into the PC, the PC instantly reclaims the Host role.

## 2. Machine Discovery and Joining
*(Source Reference: `App/Class/SocketStuff.cs`, `App/Class/TcpServer.cs`)*

MWB does not rely on complex multicast/broadcast UDP discovery out of the box. Discovery is largely manual and DNS-driven.

1. **The Listener**: Every machine always binds a `TcpListener` to TCP port `TcpPort` (default `15100`, for the clipboard server) and `TcpPort + 1` (default `15101`, for the message/input server).
2. **Pairing**: To join a new machine, the user is prompted to enter:
   - The **Security Key** (used for the AES PBKDF2 hash, typically entered via `ConnectSecurityKeyTextBox` in the UI).
   - The **Computer Name** (Hostname) of an existing machine (entered via `ConnectPCNameTextBox`).
   *(Source Reference: `SettingsXAML/Views/MouseWithoutBordersPage.xaml.cs`)*
3. **DNS Resolution**: The new machine attempts to resolve the Hostname to an IP address using standard DNS.
4. **The Handshake**: It connects to **port `TcpPort + 1` (15101)** — the message port — and sends the 64-byte `Handshake` packet containing its own Hostname and the derived Magic Number.
5. **Pool Integration**: If the handshake succeeds, both machines exchange `Hello` and `Heartbeat` packets. The new machine is registered in the `MachinePool`, and the existing machine sends a burst of `PackageType.Matrix` packets so the new machine can learn the physical screen layout.

## 3. Threading & Concurrency Model
*(Source Reference: `App/Core/Helper.cs`, `App/Core/Common.cs`)*

The protocol absolutely must not perform UI manipulations or heavy calculations inside the OS-level Mouse/Keyboard input callback threads (`InputHookProc`). Doing so will cause the OS to aggressively drop the input hooks, killing the application.

Instead, the client uses a dedicated **Helper Thread** (`Common.helper`):
1. **The `EvSwitch` Lock**: The `HelperThread` sits in a continuous `while(true)` loop, strictly blocking on an `AutoResetEvent` named `EvSwitch.WaitOne()`.
2. **The Handoff Trigger**: When a cursor hits the edge of the screen, the fast input thread updates the state machine (e.g., `MachineStuff.NewDesMachineID`) and signals `EvSwitch.Set()`.
3. **Helper Execution**: The `HelperThread` wakes up instantly. It evaluates the new `DesMachineID`:
   - If the cursor just left the machine, it invokes `Helper.MainFormDotEx(true)`, which aggressively hides the local mouse cursor and displays a fake invisible UI overlay to swallow stray local clicks.
   - If the cursor just returned to the machine, it restores the local cursor, calls `InputSimulation.MoveMouse()`, and resets all sticky keys (`InitAndCleanup.ReleaseAllKeys()`).

> [!IMPORTANT]
> A macOS client MUST replicate this asynchronous handoff model. If you attempt to manipulate `CGDisplayHideCursor()` directly inside the `CGEventTap` callback, the event tap will timeout and macOS will silently disable your global hook!

## 4. Machine Pool & Liveness (The 10-Second Rule)
*(Source Reference: `App/Class/MachinePool.cs`)*

The client maintains an internal `MachinePool` list containing `MachineInf` objects (Name, ID, Timestamp).
Because network sockets can "zombie" out without throwing exceptions (especially over Wi-Fi), the protocol relies on a strict internal **10-second Liveness Timeout** (`HEARTBEAT_TIMEOUT = 10000ms`).

1. **Keep-Alives**: Every time a machine receives a valid `DATA` payload from a socket, it updates that machine's `Time` property in the `MachinePool`.
2. **Expiration**: If a machine does not update its `Time` within 10,000ms, `MachinePool.IsAlive()` evaluates to `false`. The client considers the machine dead, nullifies its ID in the matrix, and resets the local cursor bounds.

## 5. Heartbeat vs. Awake Broadcasts
*(Source Reference: `App/Core/Common.cs`)*

To satisfy the 10-Second Rule, machines must continuously broadcast their presence.
The client implements a 1-2 second timer loop to broadcast presence to the pool (`ID.ALL`). However, it intelligently toggles the payload `PackageType`:

1. **`PackageType.Awake` (`21`)**: Sent if the machine is actively being used (i.e., `Event.InputEventCount` is incrementing) AND the desktop is currently active (not locked, no screensaver).
2. **`PackageType.Heartbeat` (`20` / `51`)**: Sent if the machine is idle, locked, or the screensaver is running.

By parsing `Awake` packets, the receiving machines can dynamically update their UI (e.g., showing a bright monitor icon vs a dimmed monitor icon in the system tray) without waiting for a socket drop.

## 6. OS Power State Resiliency
*(Source Reference: `App/Core/InitAndCleanup.cs`)*

Laptops frequently go to sleep and change networks, breaking raw TCP streams. The client actively listens to OS Power events (`SystemEvents.PowerModeChanged`):

- **Suspend (`PowerModes.Suspend`)**: The moment the OS broadcasts a sleep event, the client forcibly invokes `SocketStuff.Close()`, destroying all active TCP/Crypto streams and throwing the `MachinePool` into a disconnected state.
- **Why?**: If the sockets are kept open, the OS will freeze the TCP state. Upon waking up, the `Socket` objects will still claim `Connected == true`, causing the application to hang for several minutes waiting for TCP timeouts. By aggressively closing the sockets before sleep, the client instantly re-enters its aggressive Discovery/Handshake loop (`StartServer`) upon resume, guaranteeing a fast reconnect.

## 7. Port Binding and TCP Failover
*(Source Reference: `App/Class/TcpServer.cs`)*

When the `TcpServer` spins up, it attempts to bind to `Setting.Values.TcpPort` (Default: `15100` for clipboard, `15101` for message).
If it encounters a `WSAEADDRINUSE` (Port Already In Use) error:
1. It automatically sleeps and retries 6 times.
2. If it still fails, it spawns an invisible `netstat.exe -nao` process to scan for PIDs listening on the port.
3. If it identifies the PID, it logs the conflict for diagnostics.

> [!TIP]
> The macOS client should implement a similar port-conflict resolution. If a port is bound, gracefully log the conflicting PID using `lsof -i :<port>` or `netstat` and alert the user, rather than silently failing the listener.

## 8. Implementing a Fully Symmetrical Mac Node
*(Source Reference: `App/Core/Receiver.cs`, `App/Core/Event.cs`)*

To build a fully functional macOS implementation of MouseWithoutBorders—one that can both control and be controlled by Windows—you must replicate the complete P2P state machine. 

### Core Networking Requirements
1. **TCP Mesh**: Bind a TCP listener to port `TcpPort` (15100, clipboard) and `TcpPort + 1` (15101, message/input). Accept incoming sockets while simultaneously keeping outgoing sockets open to other machines in the Matrix.
2. **Security Handshake**: Perform the AES-256 decryption, derive the PBKDF2 keys, and validate/inject the Magic Number.
3. **Heartbeat Loop**: Broadcast `PackageType.Heartbeat` (`20`) every few minutes so the Windows machine doesn't drop the Mac from its `MachinePool`.

### Mac as the Host (Controlling Windows)
When the user physically moves the Mac's mouse across the screen border to the Windows PC, your app must transition into the Host role:
1. **Global Input Interception**: Use a macOS `CGEventTap` at the `kCGSessionEventTap` level. 
2. **Event Suppression**: When the mouse is on the Windows side, your `CGEventTap` must **swallow (drop) all physical keyboard and mouse events** so they don't affect the Mac desktop.
3. **Packet Translation**: Translate the Mac's virtual keycodes into Windows Virtual Key Codes (`wVk`) before packing them into `PackageType.Keyboard` (`122`) packets.
4. **Cursor Hiding**: Call `CGDisplayHideCursor` so the Mac user doesn't see their mouse stuck on the edge of the Mac screen while they are controlling the Windows PC.

### Mac as the Client (Being Controlled)
When the user physically moves the Windows mouse over to the Mac screen, your app transitions into the Client role:
1. **Input Injection**: Parse incoming `PackageType.Keyboard` and `PackageType.Mouse` packets and translate the Windows `wVk` back into Mac events. Inject them into the OS using `CGEventCreateKeyboardEvent` and `CGEventCreateMouseEvent`.
2. **Cursor Showing**: Ensure `CGDisplayShowCursor` is called when receiving `PackageType.NextMachine` so the Mac cursor becomes visible.
3. **Absolute Coordinates**: Map the incoming normalized `0-65535` mouse coordinates to the physical resolution of the Mac's `NSScreen.main`.
