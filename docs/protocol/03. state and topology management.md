# 03. State and Topology Management

Once the AES-256 TCP connection is established and the `Handshake` sequence finishes, MouseWithoutBorders begins tracking the health of the connection and syncing the spatial arrangement of the monitors. The protocol handles this without a centralized server, utilizing a dynamic DNS-like registry called the `MachinePool`.

## 1. ID Resolution and The Machine Pool
*(Source Reference: `App/Class/MachinePool.cs`, `App/Core/MachineStuff.cs`)*

The protocol manages network state using an extremely lightweight, highly resilient mechanism called the `MachinePool`. 

### What is a Machine ID?
A fundamental characteristic of this protocol is that a `Machine ID` (`ID` enum) is **not** a randomized GUID, a MAC address, or an IP address. 

The `Machine ID` is strictly a **1-based index (1 to 4)** representing the machine's physical slot in the `MachineMatrix` array (where slot 1 = ID 1). 
- `ID.NONE` = `0`
- `ID.ALL` = `255` (Broadcast)

Because the IDs are strictly bound to their physical layout slot, the protocol uses the `MachinePool` to dynamically map a 32-byte `MachineName` string to the `ID` integer. 
- When a machine broadcasts a packet (like a `Heartbeat`), it sets the packet's `Src` to its current Matrix ID.
- The receiving machine reads the trailing `MachineName` string from the Extended Payload, and dynamically updates its `MachinePool` registry to link that Name to that `ID`.

### MachinePool Serialization
To maintain state across reboots without a configuration server, the client serializes the `MachinePool` locally into a simple string format:
`"HostA:1,HostB:2,,"`
This is a comma-separated list of `MachineName:ID` pairs. 

### Liveness Tracking
A machine is simply considered "Alive" if either:
1. The TCP Socket is currently returning a connected status.
2. The last recorded heartbeat timestamp is within the `HEARTBEAT_TIMEOUT` window.
*(Note: The hardcoded timeout is `1,500,000` milliseconds, which equates to 25 minutes, meaning the protocol is very resilient to temporary network drops).*

### Lifecycle Packets
To keep the timestamps updated and announce `MachineName` changes, machines broadcast 64-byte Extended packets (via `SocketStuff.SendHeartBeat`):
- **`Hello` (3)**: Broadcast when the application first starts up or a new connection is manually requested. When a machine receives a `Hello`, it immediately replies with a `Heartbeat`.
- **`Heartbeat` (20)** and **`Heartbeat_ex` (51)**: Periodic signals sent to keep the timeout window active. Contains the sender's `MachineName` in the Extended Payload.
- **`Awake` (21)**: Fired when the machine wakes from sleep/hibernation to force an immediate pool update.
- **`ByeBye` (4)**: Sent during graceful shutdown to actively remove the machine from the remote pools without waiting 25 minutes for the timeout.

## 2. The Spatial Matrix Grid
*(Source Reference: `App/Core/MachineStuff.cs`)*

The "Matrix" is the core spatial logic that dictates how screens are laid out physically on the user's desk, allowing the mouse to cross screen borders seamlessly. 

The protocol supports a maximum of 4 machines (`MAX_MACHINE = 4`), stored internally as an array of 4 `MachineName` strings at indices `0, 1, 2, 3`.

### Layout Bounds and Indexing
The physical transition logic strictly uses array index lookups based on bitwise flags appended to the `PackageType.Matrix` (128) byte:

- **One Row (1x4)** (Default): 
  - Linear order: `[0] <-> [1] <-> [2] <-> [3]`
- **Wrap Around**: Indicated by `MatrixSwapFlag` (Bit 1 / Value `2`).
  - Connects the ends of the line: `[3] <-> [0]` forming a circle.
- **Two Row (2x2)**: Indicated by `MatrixTwoRowFlag` (Bit 2 / Value `4`).
  - Top Row: `[0]` (Top-Left), `[1]` (Top-Right)
  - Bottom Row: `[2]` (Bottom-Left), `[3]` (Bottom-Right)
  - Left/Right logic links: `[0]<->[1]` and `[2]<->[3]`.
  - Up/Down logic links: `[0]<->[2]` and `[1]<->[3]`.
  - (If `MatrixSwapFlag` is also set, all external edges wrap to their opposites).

### Matrix Synchronization Protocol
When a machine needs to tell other machines about a change in the physical monitor layout (e.g., the user dragged a computer icon in the UI), it does **not** send all 4 machine names in a single payload.

*(Source Reference: `SettingsXAML/Views/MouseWithoutBordersPage.xaml.cs`)*
In the UI, reordering is handled simply via dragging and dropping machine icons (which swaps indices in `MachineMatrixString` and updates `settings.json`). A file watcher detects this change, and the background service invokes the matrix update sequence.

Instead, it sends **4 separate packets consecutively**.
For `i` from `0` to `3`:
1. It instantiates a 64-byte `DATA` packet.
2. Sets `Type` to `Matrix` (plus the bitwise layout flags).
3. Sets `Src` to the 1-based index `i + 1`. (e.g., `1`, `2`, `3`, or `4`).
4. Sets the trailing `MachineName` string (bytes 32-63) to the hostname occupying that slot in the matrix.
5. Sends the packet over TCP.

**Receiving the Matrix**:
When `Receiver.cs` gets a `Matrix` packet, it extracts the index from the `Src` field and writes the `MachineName` string to `MachineMatrix[Src - 1]`. Because it expects exactly 4 packets, it does not commit the new topology to disk or calculate new edge bounds until it receives the packet where `Src == 4`.

> [!WARNING]
> **Matrix Bitmask Synchronization**
> The `PackageType.Matrix` packet defines more than just the topology array! The `Type` byte is actually a bitmask containing the user's topology settings:
> - **MatrixSwapFlag (2)**: Indicates if "Wrap Mouse Around" (`Setting.Values.MatrixCircle`) is enabled.
> - **MatrixTwoRowFlag (4)**: Indicates if "Two Row Matrix" (`Setting.Values.MatrixOneRow`) is disabled.
> 
> A receiver MUST decode these flags (e.g. `(package.Type & PackageType.MatrixSwapFlag) == PackageType.MatrixSwapFlag`) and update its local settings accordingly to prevent boundary detection desyncs!

## 3. Screen Edge Transitions & Coordinate Normalization
*(Source Reference: `App/Core/MachineStuff.cs`)*

Because an overarching matrix can consist of computers with vastly different screen resolutions, multi-monitor setups, and DPI scales, sending raw pixel coordinates over the network across screen bounds would cause the mouse to get stuck or jump erratically.

The protocol solves this via **Absolute Coordinate Normalization (`0` to `65535`)**.

When the mouse moves past the extreme edge of a screen (triggering a transition from Machine A to Machine B):
1. **Normalization (Machine A)**: The host computer calculates the absolute position of the mouse relative to its entire virtual desktop bounding box (`desktopBounds`). It maps this value to a strict `0` to `65535` range.
2. **Transport**: It sends this `65535`-normalized coordinate across the TCP socket inside the `Mouse` Payload.
3. **Re-Scaling (Machine B)**: The destination machine receives the `65535`-normalized coordinate and scales it back down to a raw pixel coordinate using its *own* local virtual desktop bounding box (`desktopBounds`).

This highly elegant mathematical normalization ensures that sliding the mouse from a 1080p laptop screen onto a 4K desktop monitor lands the cursor at the exact, mathematically proportional entry point on the edge of the new screen without requiring either computer to know the other's resolution.
