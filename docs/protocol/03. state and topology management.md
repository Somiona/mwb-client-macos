# 03. State and Topology Management

Once the AES-256 TCP connection is established and the `Handshake` sequence finishes, MouseWithoutBorders begins tracking the health of the connection and syncing the spatial arrangement of the monitors. The protocol handles this without a centralized server, utilizing a dynamic DNS-like registry called the `MachinePool`.

## 1. ID Resolution and The Machine Pool
*(Source Reference: `App/Class/MachinePool.cs`, `App/Core/MachineStuff.cs`)*

The protocol manages network state using an extremely lightweight, highly resilient mechanism called the `MachinePool`. 

### What is a Machine ID?
A fundamental characteristic of this protocol is that a `Machine ID` (`ID` enum) is **not** a randomized GUID, a MAC address, or an IP address. 

The `Machine ID` is strictly a **1-based index (1 to 4)** representing the machine's physical slot in the `MachineMatrix` array (where slot 1 = ID 1). 
- `ID.NONE` = `0`
- `ID.ALL` = `255` (Broadcast)

Because the IDs are strictly bound to their physical layout slot, the protocol uses the `MachinePool` to dynamically map a 32-byte `MachineName` string to the `ID` integer. 
- When a machine broadcasts a packet (like a `Heartbeat`), it sets the packet's `Src` to its current Matrix ID.
- The receiving machine reads the trailing `MachineName` string from the Extended Payload, and dynamically updates its `MachinePool` registry to link that Name to that `ID`.

### MachinePool Serialization
To maintain state across reboots without a configuration server, the client serializes the `MachinePool` locally into a simple string format:
`"HostA:1,HostB:2,,"`
This is a comma-separated list of `MachineName:ID` pairs. 

### Liveness Tracking
A machine is simply considered "Alive" if either:
1. The TCP Socket is currently returning a connected status.
2. The last recorded heartbeat timestamp is within the `HEARTBEAT_TIMEOUT` window.
*(Note: The hardcoded timeout is `1,500,000` milliseconds, which equates to 25 minutes, meaning the protocol is very resilient to temporary network drops).*

### Lifecycle Packets
To keep the timestamps updated and announce `MachineName` changes, machines broadcast 64-byte Extended packets (via `SocketStuff.SendHeartBeat`):
- **`Hello` (3)**: Broadcast when the application first starts up or a new connection is manually requested. When a machine receives a `Hello`, it immediately replies with a `Heartbeat`.
- **`Heartbeat` (20)** and **`Heartbeat_ex` (51)**: Periodic signals sent to keep the timeout window active. Contains the sender's `MachineName` in the Extended Payload.
- **`Awake` (21)**: Fired when the machine wakes from sleep/hibernation to force an immediate pool update.
- **`ByeBye` (4)**: Sent during graceful shutdown to actively remove the machine from the remote pools without waiting 25 minutes for the timeout.

## 2. The Spatial Matrix Grid
*(Source Reference: `App/Core/MachineStuff.cs`, `App/Core/DATA.cs`)*

The "Matrix" is the core spatial logic that dictates how screens are laid out physically on the user's desk, allowing the mouse to cross screen borders seamlessly. The protocol supports a maximum of 4 machines (`MAX_MACHINE = 4`), stored internally as an array of 4 `MachineName` strings at indices `0, 1, 2, 3`.

### Matrix Packet Structure (Type 128)

The `Matrix` packet is an **Extended Packet (64 bytes)** that conveys the name of a machine at a specific slot in the grid, along with global layout flags.

| Offset | Size | Field | Description |
| :--- | :--- | :--- | :--- |
| 0 | 1 | Type | `128` (Matrix) \| [Layout Flags](#layout-flags) |
| 1 | 1 | Checksum | Standard MWB checksum |
| 2 | 2 | Magic | Standard 24-bit magic hash (high 16 bits) |
| 4 | 4 | ID | Packet sequence ID |
| 8 | 4 | Src | Slot index (1, 2, 3, or 4) |
| 12 | 4 | Des | `ID.ALL` (0) |
| 16 | 16 | - | Unused (overlaps with input payloads, usually 0) |
| 32 | 32 | MachineName | ASCII-encoded, space-padded machine name |

#### Layout Flags

The `Type` byte (offset 0) includes bitwise flags that define the global topology configuration. These flags are typically sent in all 4 matrix packets but are authoritative in the 4th packet (`Src == 4`).

- **MatrixSwapFlag (2)**: Enabled "Wrap mouse around" (Circle mode).
- **MatrixTwoRowFlag (4)**: Enabled 2x2 layout. If absent, layout is 1x4 (one row).

Example Type values:
- `128`: 1x4 layout, no wrap.
- `130`: 1x4 layout, wrap enabled.
- `132`: 2x2 layout, no wrap.
- `134`: 2x2 layout, wrap enabled.

### Synchronization Lifecycle

#### 1. Initial Connection
When a client (e.g., macOS) connects to a host (Windows):
1. The client completes the AES handshake.
2. The client sends a `Heartbeat` or `Heartbeat_ex` (Type 51) packet.
3. The host receives this and calls `AddToMachinePool`.
4. **Broadcast**: If the host considers the machine "New" (not in its pool by name), it immediately broadcasts 4 `Matrix` packets to all connected peers.
5. **Persistence**: If the host already knew the machine, it might **not** send the matrix. The client MUST therefore persist its matrix locally and rely on it until a new matrix arrives.

#### 2. Manual Layout Change
When a user modifies the layout in the Settings UI on any machine:
1. The local `MachineMatrix` setting is updated.
2. The machine broadcasts 4 `Matrix` packets (one for each slot) to all connected peers over the TCP control channel.
3. Peers receive these packets, update their local settings, and MUST immediately refresh their crossing edge logic.

### Implementation Details

#### Receiving the Matrix
When a `Matrix` packet is received, the client extracts the index from the `Src` field and writes the `MachineName` to the corresponding slot in the machine pool. Because exactly 4 packets are expected, the client should not commit the new topology to disk or calculate new edge bounds until it receives the packet where `Src == 4`.

> [!WARNING]
> **Deduplication Warning**
> Matrix packets must have unique sequence IDs. The macOS client's `PackageDeduplicator` will drop packets if they reuse IDs from previous matrix updates within the same session. Ensure the sequence ID increments for each of the 4 packets in a set.

#### Bitmask Synchronization
A receiver MUST decode the layout flags in the `Type` byte and update its local settings (e.g., `MatrixCircle`, `MatrixOneRow`) accordingly to prevent boundary detection desyncs between machines.
