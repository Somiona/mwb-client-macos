# 06. Clipboard Sync

Synchronizing the clipboard across a network presents a unique challenge: clipboard data can range from a few bytes of text to hundreds of megabytes of raw images or files. If a large clipboard transfer were sent over the primary input socket, it would block `Keyboard` and `Mouse` packets, causing the mouse to freeze on the remote machine until the transfer finished.

To solve this, MouseWithoutBorders implements a dual-path architecture: an **In-Band Fast Path** for small data, and an **Out-of-Band Pull Model** for large data.

## 1. Local Capture and Evaluation
*(Source Reference: `App/Core/Clipboard.cs`)*

When the user copies something, the local OS triggers a `WM_CLIPBOARDUPDATE` hook. The protocol immediately reads the clipboard and evaluates it in `Clipboard.CheckClipboardEx()`:
1. **Size check**: If the data is less than `1 MB` (`MAX_CLIPBOARD_DATA_SIZE_CAN_BE_SENT_INSTANTLY_TCP`), it takes the Fast Path.
2. **Type check**: It identifies whether it is Text, an Image, or a FileDropList (File Path).
3. If the data is larger than `1 MB` (or if it's a massive file up to 100 MB), it takes the Out-of-Band Pull Path.

---

## 2. In-Band Fast Path (< 1MB)
*(Source Reference: `App/Core/Clipboard.cs`)*

For small text snippets or small images, the data is instantly streamed over the primary low-latency TCP socket. 

Because the protocol strictly uses fixed 64-byte packets (`Package.PACKAGE_SIZE_EX`) on the network, the clipboard payload must be chunked:
1. **Compression**: If the clipboard data is text/string data, the protocol compresses it using `DeflateStream` before serialization to minimize network overhead.
2. **Chunking**: The protocol uses the overlapping payload section of the `DATA` packet (Bytes 16 to 63). This provides exactly `48 bytes` (`DATA_SIZE = 48`) of payload space per packet.
3. **Transmission**: It fires a rapid sequence of 64-byte packets where:
   - `Type` is `ClipboardText` (`124`) or `ClipboardImage` (`125`).
   - The 48-byte payload contains the raw byte chunk.
4. **Termination**: When the final chunk is sent, it fires a packet with `Type = ClipboardDataEnd` (`76`).

The receiver buffers these chunks in memory. Once `ClipboardDataEnd` is received, it reconstructs the byte array and pushes it into the remote OS clipboard.

---

## 3. Out-of-Band Pull Path (> 1MB or Files)
*(Source Reference: `App/Core/Clipboard.cs`)*

If the copied data is too large, the sender does **not** push the data. Instead, it only pushes a notification.

### Step 1: The Notification
The machine that performed the copy broadcasts a lightweight `PackageType.Clipboard` (`69`) packet. This simply tells the other machines: *"I have new clipboard data, but it's large."* 
The other machines record the Sender's ID as `LastIDWithClipboardData` and show an icon, but do not download anything yet.

### Step 2: The Trigger
The actual data transfer is deferred until it is needed. It is triggered when:
- The user moves their mouse over to another machine (`PackageType.MachineSwitched` fires).
- OR the user attempts to paste.

### Step 3: The Dedicated Socket (`ClipboardAsk` & `ShakeHand`)
When triggered, the receiving machine sends a `PackageType.ClipboardAsk` (`78`) back to the host machine over the primary socket. This packet includes a `PostAction` enum (e.g., `Desktop` or `Mspaint`) inside the `Package` struct.

1. **Secondary Connection**: The receiving machine immediately spins up a secondary, dedicated TCP connection (`ConnectToRemoteClipboardSocket`) to the host machine.
2. **Secondary Handshake**: This new socket goes through its own AES-256 encryption handshake. **Crucially**, it calls `Common.SendOrReceiveARandomDataBlockPerInitialIV()` immediately upon stream creation to exchange a 16-byte random block.
3. **Header Authentication**: Instantly following the 16-byte random block, the stream sends a 64-byte `DATA` struct packet (`PackageType.ClipboardPush` or `Clipboard`) containing the `PostAction`. The receiver validates this struct to authenticate the secondary socket connection.
4. **Raw Stream Transfer**: Once the secondary socket is authenticated, the host machine dumps the massive payload (e.g., a 50 MB bitmap or file stream) directly into the TCP stream as fast as the network allows.

> [!WARNING]
> A macOS client MUST implement `Clipboard.ShakeHand()` perfectly. Creating a new `CryptoStream` is not enough. You must write the 16-byte random IV block, immediately followed by the 64-byte `DATA` struct `PackageType.Clipboard` header, otherwise the host machine will instantly reject the out-of-band file transfer socket.

Because this transfer happens on a completely separate TCP socket and thread, the primary socket remains completely unblocked, allowing the user to seamlessly move their mouse and type while the large file transfers in the background!

## 4. Large Payload Chunking Algorithm
*(Source Reference: `App/Core/Clipboard.cs`)*

Once the Out-of-Band secondary socket is established, the protocol does not simply flush the raw bytes of the file/clipboard into the TCP stream. It strictly formats the data into fixed-size structs.

The reference code defines:
- `Package.PACKAGE_SIZE_EX = 64`
- `DATA_SIZE = 48`

### The 48-Byte Payload
When transmitting a file or a large clipboard image (`SendClipboardDataUsingTCP`), the application loops through the byte array and slices it into exactly **48-byte** chunks.

1. **The Header (16 Bytes)**: The first 16 bytes of the 64-byte `DATA` struct are populated with the standard packet headers (`PackageType.ClipboardImage` or `PackageType.ClipboardText`, `Src`, `Des`, `Id`).
2. **The Payload (48 Bytes)**: The 48-byte file chunk is copied into the `DATA` struct starting at offset `16` (`PACKAGE_SIZE_EX - DATA_SIZE`).
3. **Transmission**: The 64-byte struct is encrypted and sent over the socket.
4. **Termination**: Once the entire file is sliced and transmitted, a final packet with `PackageType.ClipboardDataEnd` (`77`) is sent to inform the receiver that the file transfer is complete.

> [!WARNING]
> A macOS client receiving clipboard data MUST continuously buffer the incoming `DATA` structs, discarding the first 16 bytes of every packet, and concatenating the trailing 48 bytes into a `MemoryStream` until `ClipboardDataEnd` is received. Failure to align these 48-byte offsets will severely corrupt the incoming file.
