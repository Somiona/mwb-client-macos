# 10. IPC (Inter-Process Communication)

While the core functionality of MouseWithoutBorders operates entirely peer-to-peer over TCP sockets (`TcpPort` = 15100 for clipboard, `TcpPort + 1` = 15101 for message/input), the application architecture involves some local background communication and telemetry logging. These elements must be considered when porting the client to other operating systems like macOS or Linux.

## 1. Local IPC (Settings UI to Background Service)
*(Source Reference: `SettingsXAML/Views/MouseWithoutBordersPage.xaml.cs`, `App/Core/IpcChannelHelper.cs`)*

The reference implementation in PowerToys separates the User Interface (WinUI 3 / XAML) from the background worker process that holds the low-level OS hooks.

### Communication Mechanism
The UI does not directly invoke methods on the background process. Instead, it relies on two primary sync mechanisms:
1. **JSON Settings File Watcher**:
   When the user interacts with the UI (e.g., drags and drops a machine to reorder the matrix), the XAML ViewModel writes the new array to a local `settings.json` file. The background process uses a `FileSystemWatcher` (`Helper.GetFileWatcher`) to detect changes to this file, reloads the configuration, and triggers the matrix broadcast across the network.
2. **IPC Messages (`ShellPage.SendDefaultIPCMessage`)**:
   For immediate execution commands (such as "Generate New Key" or "Reconnect"), the UI passes an IPC message down to the PowerToys runner. While a monolithic macOS port may simply combine the UI and the service into a single process, any daemon/UI split must replicate this async message-passing approach so the UI doesn't block the daemon's input hook threads.
