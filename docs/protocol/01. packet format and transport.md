# 01. Packet Format and Transport

To implement this protocol in any language (e.g., Rust, Go, C, or Python), you must strictly adhere to the byte-level layout described below. The protocol uses fixed-length binary packets mapped directly to memory without any application-layer serialization (like JSON or Protocol Buffers).

## Endianness
All multi-byte integer values (`int32`, `uint32`, `int64`) are serialized in **Little-Endian** byte order, which matches the native architecture of Windows x86/x64 systems.

---

## Packet Size
Every single protocol packet sent over the TCP connection will be exactly one of two sizes:
- **Standard Packet**: `32 bytes` (Used for high-frequency input syncing)
- **Extended Packet**: `64 bytes` (Used for connection state, handshakes, and topology)

You must read from the TCP stream in exactly 32-byte or 64-byte chunks depending on the expected package (or dynamically check the first byte to determine the size).

---

## TCP Transport & Cryptography
*(Source Reference: `App/Class/SocketStuff.cs`, `App/Core/Encryption.cs`)*
The network transport strictly uses TCP. Every connection is completely wrapped in an AES encrypted stream.

### Port Layout
The protocol uses two TCP ports derived from a base `TcpPort` (default `15100`):

| Port | Purpose | Server Handler | Client Connects To |
| :--- | :--- | :--- | :--- |
| `TcpPort` (15100) | **Clipboard** server | `AcceptConnectionAndSendClipboardData` | `ConnectToRemoteClipboardSocket` |
| `TcpPort + 1` (15101) | **Message/Input** server | `TCPServerThread` → `MainTCPRoutine` | `StartNewTcpClientThread` → `MainTCPRoutine` |

### 1. Key Derivation (PBKDF2)
The raw security key string entered by the user is derived into a 32-byte AES key using `PBKDF2` (`HMAC-SHA512`) with **50,000 iterations** (`Encryption.GenLegalKey()`). 
- **Salt Quirks**: The Salt used for PBKDF2 is the string `"18446744073709551615"` (the string representation of `UInt64.MaxValue`). **Crucially**, it is encoded in **UTF-16LE** via the `Common.GetBytesU()` method (which uses `.NET`'s `Encoding.Unicode`). Failure to encode the salt as UTF-16LE will result in incorrect AES keys.

### 2. Initialization Vector (IV)
The connection uses AES-256 in **CBC (Cipher Block Chaining)** mode, with **Zeros** padding and a block size of 16 bytes (`Encryption.InitEncryption()`).
- **IV Quirks**: The base Initialization Vector is the exact 16-byte string `"1844674407370955"`. **Crucially**, unlike the salt, the IV is encoded in **ASCII** via the `Common.GetBytes()` method. 

### 3. The 16-Byte Handshake Shift (Important!)
To prevent known-plaintext attacks on the first protocol packet, the implementation shifts the CBC cipher block state immediately after opening the `CryptoStream`. 
- **Both the Client and the Server** immediately generate exactly 16 bytes of cryptographically random data and write it to the TCP socket.
- **Both sides** must also read exactly 16 bytes of this random data from the socket before attempting to send or receive any structured protocol packets.

### 4. Dispatch Loop
The receiving loop continually reads the decrypted stream. 
- It reads the first byte to determine `PackageType` and expects either 32 or 64 total bytes.
- It validates the `Checksum` and `Magic Number` (see Header below).
- It validates the sequence `Id` against a circular array of the last 50 processed IDs to drop network duplicates.
- It routes the struct to the specific handler based on `PackageType`.

## The Header (Bytes 0 - 15)
*(Source Reference: `App/Core/DATA.cs`, `App/Class/SocketStuff.cs`)*
Every packet (both 32-byte and 64-byte) shares an identical 16-byte header.

| Offset | Length (Bytes) | Type | Field | Description |
| :--- | :--- | :--- | :--- | :--- |
| `0` | 1 | `uint8` | `PackageType` | The primary identifier of the packet payload (see Enum section below). |
| `1` | 1 | `uint8` | `Checksum` | An 8-bit checksum. It is calculated by adding up all bytes from offset `2` to `31` (modulo 256). **Note**: The checksum *only* covers the first 32 bytes, even for 64-byte Extended packets! |
| `2` | 1 | `uint8` | `Magic (Low)` | The lower 8 bits of the top 16 bits of the 24-bit derived Magic Hash. `(MagicHash >> 16) & 0xFF` |
| `3` | 1 | `uint8` | `Magic (High)`| The upper 8 bits of the top 16 bits of the 24-bit derived Magic Hash. `(MagicHash >> 24) & 0xFF` |
| `4` | 4 | `int32` | `Id` | An incrementing sequence number. Used by the receiver to drop duplicate packets. |
| `8` | 4 | `uint32` | `Src` | Sender Machine ID. |
| `12` | 4 | `uint32` | `Des` | Destination Machine ID (`255` is used for `ID.ALL` broadcasts). |

*Note: Before passing the payload up the stack to application handlers, bytes 1, 2, and 3 are zeroed out by the receiver's validation step.*

## Overlapping Payload (Bytes 16 - 31)
*(Source Reference: `App/Core/DATA.cs`)*
Starting at byte offset 16, the data layout depends entirely on the `PackageType` defined at byte 0. Because the C# implementation uses `[StructLayout(LayoutKind.Explicit)]`, these structures completely overlap in memory. This means `Kd`, `Md`, `Machine1-4`, and `PostAction` all start at byte offset 16 (`FieldOffset(16)`).

### 1. Keyboard Payload (`PackageType = 122`)
| Offset | Length (Bytes) | Type | Field | Description |
| :--- | :--- | :--- | :--- | :--- |
| `16` | 8 | `int64` | `DateTime` | Sender's timestamp ticks (`DateTime.Now.Ticks / 10000`). |
| `24` | 4 | `int32` | `wVk` | Virtual Key Code of the keystroke. |
| `28` | 4 | `int32` | `dwFlags` | Key state flags (e.g., KeyUp vs KeyDown). |

### 2. Mouse Payload (`PackageType = 123` or `121`)
| Offset | Length (Bytes) | Type | Field | Description |
| :--- | :--- | :--- | :--- | :--- |
| `16` | 4 | `int32` | `X` | Relative or scaled Absolute X coordinate. |
| `20` | 4 | `int32` | `Y` | Relative or scaled Absolute Y coordinate. |
| `24` | 4 | `int32` | `WheelDelta` | Scroll wheel delta. |
| `28` | 4 | `int32` | `dwFlags` | Mouse event flags (e.g., Left Down, Move). |

### 3. Handshake and HandshakeAck Payload (`PackageType = 126, 127`)
| Offset | Length (Bytes) | Type | Field | Description |
| :--- | :--- | :--- | :--- | :--- |
| `16` | 16 | `byte[]` | `Challenge` | 16 bytes of cryptographically random data used for the Challenge-Response mechanism. |

### 4. Heartbeat Payload (`PackageType = 20, 51, etc.`)
Heartbeat packets **do not** populate any specific fields in the 16-31 offset range (they default to `0`). The payload of a Heartbeat is simply the standard 16-byte header combined with the 32-byte Extended Payload containing the Sender's `MachineName`.

### 5. Matrix Topology Payload (`PackageType = 128`)
Like Heartbeat packets, Matrix packets **do not** populate any specific fields in the 16-31 offset range (they default to `0`). The payload of a Matrix packet relies purely on the 32-byte Extended Payload containing a `MachineName` string.

### 6. Clipboard Initial Payload (`PackageType = 69, 79`)
Used to establish a secondary stream connection before transferring the file payload.
| Offset | Length (Bytes) | Type | Field | Description |
| :--- | :--- | :--- | :--- | :--- |
| `16` | 4 | `int32` | `PostAction` | Action code for the clipboard transfer (e.g., `Desktop`, `Mspaint`, `Other`). |
| `20` | 12 | `byte[]` | `Padding` | Unused zeros to pad out to 32 bytes. |

---

## Extended Payload (Bytes 32 - 63)
*(Source Reference: `App/Core/DATA.cs`)*
If the packet is a 64-byte `Extended` package (determined by the `PackageType`), the final 32 bytes always contain the Sender's Machine Name.

| Offset | Length (Bytes) | Type | Field | Description |
| :--- | :--- | :--- | :--- | :--- |
| `32` | 32 | `ASCII String`| `MachineName` | 32-byte ASCII string representing the host name. It is strictly padded with spaces (`0x20`) if the name is shorter than 32 characters. |
