# 07. Drag and Drop File Transfer

Drag and Drop across network boundaries is notoriously difficult because local operating systems (like Windows and macOS) rely on local memory pointers and COM/OLE objects for drag operations. MouseWithoutBorders solves this using a complex 12-step state machine that effectively "fakes" the drag operation on both sides of the network.

## 1. Drag Detection (Phase 1)
*(Source Reference: `App/Core/DragDrop.cs`)*

The protocol cannot simply hook OLE events globally without blocking the OS. It must mathematically infer a drag operation based on mouse state and screen borders.

1. **Mouse Down (`Step01`)**: The global hook on the Source Machine detects `WM_LBUTTONDOWN` and sets a `MouseDown = true` flag (tracked via the `DragDrop.cs` state machine). At this point, it does not actually know if a file is being dragged or if the user is just highlighting text.
2. **Border Crossing (`Step02`)**: The user drags the mouse to the screen edge, triggering a handover to the Destination Machine (as detailed in `05. cross border handle.md`).
3. **The Inquiry (`Step03`)**: Because the Destination Machine receives the cursor while the mouse button is held down, the Destination Machine fires a `PackageType.ExplorerDragDrop` (`72`) packet *back* to the Source Machine. It is essentially asking: *"You just handed over the cursor with the mouse button down. Are you dragging a file?"*
4. **OS Interception (`Step04-05`)**: To figure out *what* the user is dragging, the Source Machine forcefully spawns an invisible "Helper Form" directly underneath the mouse cursor's last known location. Because the mouse is currently holding an OLE drag object, this invisible form instantly receives the OS's `DragEnter` event, allowing the protocol to extract the file path!

## 2. Visual Transition (Phase 2)
*(Source Reference: `App/Core/DragDrop.cs`)*

Once the Source Machine extracts the file path, it must convince the user that the file has successfully crossed the border.

1. **Broadcast (`Step06`)**: The Source Machine sends `PackageType.ClipboardDragDrop` (`70`) and `PackageType.ClipboardDragDropOperation` (`75`) to the Destination Machine.
2. **Target Visuals (`Step08-09`)**: The Destination Machine sets an internal `IsDropping = true` flag. Because the destination OS doesn't actually have a file in its drag queue, the protocol forcefully renders a custom floating "Drop Icon" at the local mouse cursor's `(X,Y)` coordinates. As the user moves the mouse around the destination screen, this icon follows it, giving the visual illusion that the file is being dragged.

## 3. Drop and Transfer (Phase 3)
*(Source Reference: `App/Core/DragDrop.cs`)*

The user finally decides where to drop the file on the new machine and releases the physical mouse button.

1. **Mouse Up (`Step10`)**: The Destination Machine's hook detects `WM_LBUTTONUP`. It immediately hides the fake floating icon.
2. **The Pull Request**: Because a dragged file is typically large, the protocol completely reuses the **Out-of-Band Pull** architecture detailed in `06. clipboard sync.md`.
3. It calls `Clipboard.GetRemoteClipboard("desktop")`, which sends a `PackageType.ClipboardAsk` (`78`) over the main socket, spins up a *secondary* TCP connection, and downloads the file contents in the background as a raw encrypted stream.
4. **File Placement**: The file is saved directly to the user's Desktop in a folder named `MouseWithoutBorders`. The protocol then triggers a Shell execute command to pop the folder open, revealing the dropped file to the user.

## 4. Cancellation
*(Source Reference: `App/Core/DragDrop.cs`)*

If the user drags the file across the border, but then changes their mind and drags it back to the original screen without letting go of the mouse button (`Step11`), the Source Machine detects the reversal and sends `PackageType.ClipboardDragDropEnd` (`71`). The Destination Machine simply destroys the floating icon and resets its state machine.
